# 배경화면 모드 기술 스파이크

## 목적

배경화면 모드는 제품 정체성에 중요하지만 운영체제, GPU, 입력 처리와 성능 위험이 큽니다. 전체 게임 구현 전에 작은 Godot 프로젝트로 기술 가능성을 검증합니다.

## 검증 대상

### 창 동작

- 투명 배경
- 테두리 없는 창
- 항상 위 또는 바탕화면 레이어 동작
- 작업 표시줄 표시 여부
- 포커스를 가져오지 않음
- 마우스 입력 통과
- 지정 단축키 또는 클릭으로 인터랙션 전환
- 일반 창으로 복귀

### 화면 환경

- 단일 모니터
- 다중 모니터
- 서로 다른 해상도 조합
- Windows 화면 배율 100%, 125%, 150%
- 화면 잠금과 절전 후 복귀
- 창 위치와 크기 저장

### 그래픽

- 투명 PNG 도트 캐릭터 표시
- nearest-neighbor 확대
- 정수 배율 유지
- 투명 경계의 색 번짐 여부
- GPU별 투명 창 표시

### 성능

- 생물 10, 30, 50개 표시
- Idle과 이동 애니메이션 혼합
- 비활성 사육장 업데이트 제한
- CPU와 GPU 사용량
- 메모리 사용량
- 장시간 실행 안정성

## 스파이크 범위

게임 시스템은 구현하지 않습니다.

포함:

- 단일 투명 창
- 테스트 생물 1종
- Idle과 이동
- UI 표시·숨김
- 입력 통과 전환
- 모니터 선택
- FPS·CPU 측정용 디버그 정보

제외:

- 실제 환경 시뮬레이션
- 합사와 포식
- 저장 데이터 전체 구조
- 최종 UI 디자인
- 실제 설치 프로그램

## 구현 후보 비교

| 후보 | 설명 | 장점 | 위험 |
|---|---|---|---|
| 투명 오버레이 창 | 바탕화면 위에 투명 창을 표시 | Godot 내부 구현 집중 | 창 계층, 포커스, GPU 호환성 |
| 전용 라이브 월페이퍼 연동 | 외부 런타임 또는 플랫폼 활용 | 바탕화면 통합 가능 | 배포 의존성과 설정 복잡도 |
| 일반 무테 창 | 바탕화면 크기의 일반 창 | 구현 단순 | 실제 배경화면 경험과 차이 |

MVP 권장 시작점은 투명 오버레이 창이며, 실패 시 일반 무테 관찰 모드를 대체안으로 유지합니다.

## 테스트 시나리오

1. 앱 실행 후 투명 배경에 생물만 표시
2. 다른 앱을 클릭해도 포커스가 이동하지 않음
3. 평상시 마우스 입력이 아래 창으로 전달됨
4. 단축키로 인터랙션 모드 진입
5. 생물을 클릭하고 작은 패널 표시
6. 다시 입력 통과 모드로 복귀
7. 모니터 변경과 화면 배율 변경
8. 절전 후 복귀
9. 4시간 이상 연속 실행

## 통과 기준

- 주요 테스트 PC에서 투명 표시가 깨지지 않음
- 입력 통과와 인터랙션 전환이 예측 가능함
- 포커스 도난으로 업무 흐름을 방해하지 않음
- 기본 생물 수에서 성능 예산을 충족함
- 일반 플레이와 동일한 상태 데이터를 표시할 수 있는 구조가 확인됨
- 실패 시 일반 무테 관찰 모드로 전환 가능한 대체 경로가 있음

- 항상 위 동작을 허용할지
- 인터랙션 진입 단축키
- 성능 예산과 최소 지원 PC 사양
